**ADR 2025-11.7 — Выбор технологического процесса разработки**

**Дата:** 8 ноября 2025 г.  
**Статус:** Принято

**Контекст:**  
Проект реализуется командой малой/средней величины, система микросервисная, требуется быстрый вывод фич, частые правки в проде и гибкое распределение приоритетов (инциденты, регуляторные правки, фичи). Нужна методика, минимизирующая административные церемонии и поддерживающая непрерывную поставку.

**Рассмотренные варианты:**  
1. **Waterfall:** жёсткие фазы, прогнозируемость, но медленные итерации.  
2. **Scrum:** фиксированные спринты, планирование и ретроспективы; хорош для предсказуемого бэклога.  
3. **Scrumban:** гибрид Scrum/Kanban, добавляет ритмы Scrum к Kanban‑потоку.  
4. **Kanban:** непрерывный поток задач, WIP‑лимиты, минимальные церемонии.

**Решение:**  
Выбран Kanban.

**Обоснование:**  
- Подходит для переменчивых приоритетов (инциденты, срочные правки, регуляторика).  
- Поддерживает непрерывную доставку и быстрые релизы (CI/CD).  
- Минимум ритуалов — команда может сосредоточиться на скорости решения задач.  
- WIP‑лимиты и визуализация потока помогают выявлять узкие места и улучшать процесс.  
- Легче масштабировать на несколько маленьких команд и интегрировать операционные задачи.

**Практики и правила (рекомендуемые):**  
- Доска: колонки Backlog → Ready → In Progress → Code Review → QA → Staging → Done (+ Blocked).  
- Введение WIP‑лимитов для колонок In Progress / Code Review / QA.  
- Pull‑based workflow: разработчик тянет задачу из Ready.  
- Definition of Ready / Definition of Done — явно прописанные критерии.  
- Классы обслуживания: Standard, Expedite (hotfix), Fixed‑date. Правила SLA для Expedite.  
- Политика работы с блокерами: пометка, ответственный, SLA на разблокирование.  
- Регулярные короткие встречи: ежедневный Standup (10–15 мин), еженедельный sync по приоритетам, ежемесячная ретроспектива.  
- Метрики: Cycle Time, Lead Time, Throughput, WIP, Cumulative Flow Diagram. Целевые значения и пороги для алертов.  
- Автоматизация: интеграция доски с CI (PR → переход состояния), автоматическое закрытие при merge/deploy.

**Риски и меры смягчения:**  
- Риск: потеря предсказуемости командной загрузки. Митигировать — установить SLA/expectations для планирования релизов и использовать еженедельное планирование.  
- Риск: бесконечные приоритетные вмешательства (context switching). Митигировать — строгие WIP‑лимиты и классы обслуживания.  
- Риск: недостаток дисциплины → рост WIP. Митигировать — ретроспективы, метрики и ответственность владельцев процессов.

**Шаги внедрения:**  
1. Выбрать инструмент (Jira/Azure Boards/GitHub Projects/Trello) и настроить доску по рекомендованной схеме.  
2. Прописать Definition of Ready/Done, правила переходов и WIP‑лимиты.  
3. Настроить интеграцию с CI/CD и PR‑workflow.  
4. Обучить команду правилам Kanban и провести пилот (1–2 месяца).  
5. Собирать метрики, проводить ежемесячные ретроспективы и корректировать WIP/процесс.

**Последствия:**  
- Меньше ритуалов, более быстрый цикл доставки, требуется дисциплина и мониторинг метрик.  
- Потребность в прозрачных SLA/expectations для планирования релизов.  
- Необходима инвестиция в автоматизацию и регулярный анализ метрик.
